1. Executive Summary

The analyzed binary is a Linux ELF 64-bit malware sample that functions as a UDP-based backdoor. Upon execution, it attempts to disguise its process name as [syslog] to blend in with legitimate system processes. It establishes a listening UDP socket to receive encrypted commands from a remote controller. The malware supports updating its encryption keys, writing file data to the local system (specifically to a file named system.conf), and likely executing remote shell commands via a remoteExec function. Communication is obfuscated using a simple XOR cipher and involves raw packet manipulation to calculate custom UDP checksums. Given its capabilities for remote command execution and file manipulation, the threat level is High.

2. Malware Capabilities

Identified capabilities and behaviors exhibited by this malware sample.
  • Process Masquerading: Renames the running process to [syslog] to avoid detection in process lists.
  • C2 Communication: Listens on a UDP socket (associated with magic port 0x84b) for incoming packets.
  • Cryptographic Obfuscation: Uses a 4-byte XOR key (idkey) to encrypt and decrypt command payloads.
  • File Operations: Capable of writing received file chunks to system.conf on disk.
  • Remote Execution: Contains a remoteExec function to run shell commands via popen and return the output to the C2 server.
  • Anti-Analysis/Debugging: Ignores SIGCHLD (signal 17) signals to prevent zombie processes and handle child termination silently (common in forking shells).

4. Technical Analysis

In-depth technical examination of the malware code structure, algorithms, and implementation details.
Process Name Spoofing

The main function checks if the argument count (argc) is greater than 8. If so, it overwrites argv[0] with the string "[syslog]" using hardcoded hex values (0x7379735b, 0x64676f6c, 0x5d). This technique hides the malware's true identity in tools like top or ps.

C2 Communication & Protocol

The malware creates a UDP socket (socket(2,3,0x11)). It enters a loop receiving data into a buffer. It validates packets by checking for a specific magic number 0x84b (decimal 2123) and a hard-coded idkey. If valid, it decrypts the payload.

Command Handling

Based on the decompilation, the malware interprets command types located at offset local_18[9]:

  • Type 0x01: Updates the idkey used for encryption.
  • Type 0x02: Writes data to system.conf.
