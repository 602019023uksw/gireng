1. Executive Summary

The binary is a Google Sheets-based backdoor written in C that targets Linux platforms. The malware collects comprehensive system information including hostname, IP addresses, OS version, username, current working directory, and environment variables, then sends this data to a C2 server via Google Sheets API. It supports remote command execution (C-C), file upload (C-U), and file download (C-D) operations. All C2 communication occurs over HTTPS to sheets.googleapis.com using OAuth2 Bearer tokens. The malware uses zlib compression for data transmission and implements a chunking mechanism (45000 bytes per chunk) for large data transfers. To prevent multiple instances, it attempts to read a configuration file at /tmp/kworofd.cfg. The malware is heavily obfuscated with XOR encoding, Base64 encoding, and control flow manipulation techniques.

3. Technical Analysis

In-depth technical examination of the malware's code structure, algorithms, and implementation details.

The malware begins execution by calling __libc_start_main which transfers control to FUN_004095c0 (main function). The main function first calls FUN_00408190 which appears to initialize the runtime environment and check for configuration files. It then calls FUN_00408960 to initialize internal data structures and FUN_004094b0 for additional setup.

The malware collects system information through FUN_00408b20 which gathers hostname using gethostname(), network interfaces and IP addresses using getifaddrs() and getnameinfo(), OS information using uname(), username via getpwuid(getuid()), current working directory using getcwd(), and the LANG environment variable using getenv(). This information is formatted into a structured string and encrypted using FUN_00407f50.

The core C2 communication occurs in FUN_004093b0 which builds an HTTP POST request to sheets.googleapis.com. The request includes an OAuth2 Bearer token stored in DAT_00763cf0 and a spreadsheet ID in DAT_00763d10. The data payload is compressed using zlib (FUN_00407d90) before transmission. The malware uses chunking for large data transfers, splitting data into 45000-byte segments labeled with identifiers like "a2", "a3", etc.

Command processing in FUN_004095c0 handles three command types: "C-C" for command execution via popen(), "C-U" for file upload using fopen()/fwrite(), and "C-D" for file download. The malware parses command strings with format "C-C-" or "C-U-" and executes accordingly. For command execution, it uses popen() to execute shell commands and returns the output. For file operations, it reads/writes files and transmits the data through the C2 channel.

The encryption function FUN_00407d90 uses zlib compression (version 1.1.4) and appears to implement additional encoding layers. The error message "Error no key path: Operation not permitted" indicates the malware expects a configuration file at /tmp/kworofd.cfg which contains encryption keys or authentication tokens.

The malware implements a polling loop with sleep intervals, initially sleeping 1 second between iterations, then increasing to random intervals between 300-789 seconds (5-13 minutes) after 121 iterations, providing basic anti-analysis capabilities through timing variation.

6. Operational Flow

  1. Initialization: The binary starts at entry point, calls __libc_start_main which transfers to FUN_004095c0 (main). It initializes runtime environment and data structures.
  2. System Information Collection: Calls FUN_00408b20 to gather comprehensive system data including hostname, IP addresses (via getifaddrs), OS version (uname), username (getpwuid), current directory (getcwd), LANG environment variable, and timestamp. This data is encrypted and formatted.
  3. C2 Registration: Calls FUN_004093b0 to send collected system information to Google Sheets API. The request uses OAuth2 Bearer token authentication and targets the spreadsheet endpoint /v4/spreadsheets/%s/values:batchUpdate.
  4. Command Polling Loop: Enters an infinite loop that:
    • Calls FUN_00409300() to retrieve commands from Google Sheets
    • Parses command strings in format "C-C-" (execute), "C-U-" (upload), or "C-D-" (download)
    • Executes commands using popen() for execution or file I/O operations
    • Returns results to C2 via Google Sheets API
    • Sleeps 1 second between iterations (with randomization after 121 iterations)
  5. Data Transmission: All data is compressed using zlib (FUN_00407d90) before transmission. Large data is chunked into 45000-byte segments with labels like "a2", "a3", etc.
  6. Anti-Analysis: Implements timing variation (sleep randomization) and checks for configuration file at /tmp/kworofd.cfg. The malware is heavily obfuscated with XOR encoding, Base64 encoding, and control flow manipulation to evade static analysis.

11. Conclusion

This malware is a sophisticated Linux backdoor that uses Google Sheets as a covert C2 channel. The threat level is HIGH due to its multi-function capabilities, strong obfuscation, and use of legitimate cloud services for command and control. The malware can execute arbitrary commands, upload/download files, and collect sensitive system information. While it requires OAuth2 credentials to function, if these are compromised, the attacker gains full remote control over the infected system. The use of Google Sheets API makes detection difficult as the traffic appears legitimate. Organizations should monitor for suspicious Google API usage and implement network controls to restrict unauthorized API access.
