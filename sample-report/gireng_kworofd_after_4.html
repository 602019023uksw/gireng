1. Executive Summary

This binary is a Linux malware specimen that utilizes Google Sheets as a Command and Control (C2) infrastructure to evade detection via "Living off the Land" techniques. Upon execution, the malware decrypts configuration data (likely credentials and spreadsheet IDs) stored within its own binary using AES-128-CBC. It authenticates to Google Cloud Platform using OAuth2 with a JWT (JSON Web Token) signed by an RSA key found in the decrypted configuration. Once authenticated, it performs system reconnaissance—gathering hostname, IP addresses, OS information, and user details—and uploads this data to a specific Google Sheet. The malware then enters a persistent polling loop, reading commands from the spreadsheet to execute arbitrary shell commands, exfiltrate files, or write files to the local disk. The binary statically links OpenSSL for cryptographic operations and network communication over HTTPS.

4. Technical Analysis

In-depth technical examination of the malware code structure, algorithms, and implementation details.
[Configuration Loading]

The malware begins by reading its own binary path via /proc/self/exe to access embedded, encrypted configuration data. It opens the binary file and reads chunks of data into memory. It then uses AES-CBC decryption with hardcoded keys and IVs found within the code (e.g., strings like mNNMyNvSkC2SMSg24Rw5ekigOILRaS6_h7fnSfHpsbucOH7VdZVnVRvhrFSCC_0bbPzIRSyPR2mIOfC_D8AA) to decrypt this data into global variables representing the Spreadsheet ID, OAuth scopes, and RSA keys.

Code Evidence (fcn.00408190 @ 0x00408190): 
rax = readlink ("/proc/self/exe", rsp, 0x1000);
// ...
rax = fopen (rbx, 0x4f479d); // "rb"
// ...
AES_cbc_encrypt ();
[Authentication with Google Cloud]

The malware constructs a JSON Web Token (JWT) using the decrypted RSA key. It sets the audience to https://oauth2.googleapis.com/token and includes scopes for Google Spreadsheets and Drive. It then sends a POST request to the OAuth2 token endpoint to exchange the signed JWT for an OAuth2 Bearer token.

Code Evidence (fcn.00408960 @ 0x00408960): 
snprintf (rbx, 0x1000, "{\"aud\":\"https://oauth2.googleapis.com/token\",\"exp\":%s,\"iat\":%s,\"iss\":\"%s\",\"sub\":\"%s\",\"scope\":\"https://www.googleapis.com/auth/spreadsheets.readonly https://www.googleapis.com/auth/spreadsheets https://www.googleapis.com/auth/drive.file...}", r12);
// ...
snprintf (r13, rbx, "POST %s HTTP/1.1\r\nHost: oauth2.googleapis.com\r\n...Grant-Type: jwt-bearer...", "/token");
[System Reconnaissance]

Before entering the command loop, the malware gathers detailed system information including the hostname, network interface IP addresses (using getifaddrs), OS version (uname), current user ID (getuid/getpwuid), and current working directory (getcwd). This data is formatted into a string to be uploaded to the C2.

Code Evidence (fcn.00408b20 @ 0x00408b20): 
eax = gethostname ();
// ...

// ...
eax = uname ();
// ...
eax = getcwd ();
[C2 Command Protocol]

The malware communicates by reading specific cells in a Google Sheet. Commands follow a specific syntax parsed by splitting the input string on the hyphen (-) character.

  • Format: S-{Command}-{Argument}
  • Command S-C: Execute shell command. The argument is the command string. The output is captured and uploaded.
  • Command S-D: Download/Read file. The argument is the local file path. The file content is read, Base64 encoded, and uploaded.
  • Command S-U: Upload/Write file. The argument is the local file path. Data (presumably from the sheet) is Base64 decoded and written to this path.
    • Code Evidence (main @ 0x004095c0): 
rax = strchr (rbx, 0x2d); // split by '-'
// ...
eax = strcasecmp (rbx, 0x5124e4); // Check if starts with "S"
// ...
eax = strcasecmp (r12, 0x5124e4); // Check second part "C"
// ...
rax = popen (rbx, 0x4f6333); // Execute command

7. Operational Flow

  1. Initialization: The binary resolves its own path (/proc/self/exe) and reads encrypted configuration blobs. — Evidence: fcn.00408190 @ 0x00408190
  2. Setup: The configuration is decrypted using AES-CBC, revealing the Google Service Account credentials and the Spreadsheet ID. — Evidence: fcn.00408190 @ 0x00408190
  3. Authentication: A JWT is signed with the loaded RSA key and exchanged for an OAuth2 Bearer token via oauth2.googleapis.com. — Evidence: fcn.00408960 @ 0x00408960
  4. Reconnaissance: System details (Hostname, IPs, User, Path, OS) are gathered. — Evidence: fcn.00408b20 @ 0x00408b20
  5. C2 Registration: The reconnaissance data is Base64 encoded and uploaded to the Google Sheet. — Evidence: fcn.004093b0 @ 0x004093b0
  6. Command Loop: The malware enters a while(1) loop:
    • Sleeps for a random interval.
    • Reads a specific cell from the Sheet to check for commands. — Evidence: fcn.00409110 @ 0x00409110
    • If a command is found, parses the format S-{Type}-{Arg}.
    • Executes the command (Shell exec, File Read, or File Write).
    • Uploads the result/output back to the Sheet.
    • Clears the command cell to prevent re-execution. — Evidence: fcn.004094b0 @ 0x004094b0

8. C2 & Networking

  • C2 Servers: oauth2.googleapis.com, sheets.googleapis.com — Evidence: Strings list
  • Protocols: HTTPS (Google APIs) — Evidence: fcn.00408960 uses Host: oauth2.googleapis.com and SSL function imports.
  • Communication Pattern: Polling mechanism via Google Sheets API. The malware acts as a client interacting with a legitimate cloud service to blend in with normal traffic.
  • Command Format:
  • S-C-{command}: Execute shell command.
  • S-D-{filepath}: Read file and exfiltrate.
  • S-U-{filepath}: Write file from C2 data.
  • Authentication: OAuth2 Bearer token obtained via JWT assertion (Service Account flow).
  • Data Format: JSON (for API payloads), Base64 (for encoding command/file data in sheet cells).

9. Recommendations

1
Block or monitor network traffic to oauth2.googleapis.com and sheets.googleapis.com if not authorized for business use, specifically looking for JWT assertion grant types (urn:ietf:params:oauth:grant-type:jwt-bearer).
2
Scan systems for binaries that read /proc/self/exe and contain references to Google Sheets API endpoints, as this is a known evasion technique.
3
Identify the compromised Google Service Account associated with the Spreadsheet ID (found in the decrypted config) and revoke its access immediately.
4
Investigate the contents of the Google Sheet used as a C2 to identify exfiltrated data, executed commands, and other compromised hosts.
5
Monitor for the specific User-Agent strings used by the malware: Directory API Google-API-Java-Client/2.0.0 and Google-HTTP-Java-Client/1.42.3.
